## 2025-03-09 - Accessibility of Icon-Only Buttons
**Learning:** This application heavily utilizes custom icon-only buttons (using `lucide-react`) across key views (e.g., layout navigation, adding an item, item details). By default, these buttons lacked proper ARIA labels and keyboard focus states (`focus-visible`), which makes the experience poor for screen reader users and keyboard navigators.
**Action:** When working on future components or refining existing ones in this design system, always ensure that any icon-only button is accompanied by an `aria-label` (localized in Spanish per conventions) and explicitly receives focus states (`focus-visible:ring-2 focus-visible:ring-primary focus-visible:outline-none`).## 2024-05-18 - Form Accessibility Pattern
**Learning:** The AddItem form used standard Tailwind classes for styling but lacked explicit `id` and `htmlFor` pairings, which are required for screen reader accessibility and improving the clickable area of inputs. The custom "Barter Auction" toggle switch (`.sr-only peer` pattern) specifically required `peer-focus:ring-2 peer-focus:ring-primary` on its visual element to ensure keyboard navigability, and its descriptive text needed to be wrapped in a `<label htmlFor="...">` to be clickable.
**Action:** When creating or modifying forms, always ensure every input has an explicit `id` matched to a `<label htmlFor="id">`. For custom toggles using the `peer` class, always verify that keyboard focus states (`peer-focus:*`) are implemented on the visual indicator and that surrounding text is wrapped in the corresponding label.

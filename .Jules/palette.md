## 2025-03-09 - Accessibility of Icon-Only Buttons
**Learning:** This application heavily utilizes custom icon-only buttons (using `lucide-react`) across key views (e.g., layout navigation, adding an item, item details). By default, these buttons lacked proper ARIA labels and keyboard focus states (`focus-visible`), which makes the experience poor for screen reader users and keyboard navigators.
**Action:** When working on future components or refining existing ones in this design system, always ensure that any icon-only button is accompanied by an `aria-label` (localized in Spanish per conventions) and explicitly receives focus states (`focus-visible:ring-2 focus-visible:ring-primary focus-visible:outline-none`).## 2025-03-09 - Form Input Accessibility (Labels & Custom Toggles)
**Learning:** In the Truekio design system, form inputs and custom toggle switches (using the `.sr-only peer` pattern) often lack explicit `id` attributes and corresponding `htmlFor` mappings on their descriptive `<label>` tags. This creates significant accessibility barriers for screen readers and breaks the standard UX expectation that clicking a label focuses its associated input. Furthermore, custom visually-styled toggles using the peer pattern frequently lack visual focus indicators (`focus-visible` state) when navigated via keyboard.
**Action:** Always ensure every form input (`<input>`, `<select>`, `<textarea>`) has a unique `id` that exactly matches the `htmlFor` attribute of its associated `<label>`. For custom toggle designs relying on a visually hidden input (`.sr-only peer`), you must add visual focus classes (e.g., `peer-focus-visible:ring-2 peer-focus-visible:ring-primary`) to the styling `<div>` so keyboard users can track their position within the form.

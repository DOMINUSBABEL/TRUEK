1. Identify performance bottleneck:
In `src/pages/Trades.tsx` and `src/pages/ItemDetail.tsx`, the `acceptOffer` and `handleAccept` functions execute multiple `updateDoc` calls sequentially, creating a "waterfall" of network requests.
Additionally, when rejecting other trades, they use a `for...of` loop with `await updateDoc` for each trade.

For `Trades.tsx` (`handleAccept`):
```javascript
// Update trade status
await updateDoc(doc(db, 'trades', trade.id), { status: 'accepted' });

// Update items status
await updateDoc(doc(db, 'items', trade.targetItemId), { status: 'traded' });
await updateDoc(doc(db, 'items', trade.offeredItemId), { status: 'traded' });
```
and then a loop to reject others:
```javascript
for (const tDoc of snapshot.docs) {
  // ...
  await updateDoc(doc(db, 'trades', tData.id), { status: 'rejected' });
}
```

For `ItemDetail.tsx` (`acceptOffer`):
```javascript
// Update accepted trade
await updateDoc(doc(db, 'trades', tradeId), { status: 'accepted' });

// Reject other trades
const otherOffers = auctionOffers.filter(o => o.id !== tradeId);
for (const offer of otherOffers) {
  await updateDoc(doc(db, 'trades', offer.id), { status: 'rejected' });
}

// Update items status
await updateDoc(doc(db, 'items', item.id), { status: 'traded' });
await updateDoc(doc(db, 'items', offeredItemId), { status: 'traded' });
```

This takes N+3 sequential network round-trips.

2. Optimization:
We can use Firestore's `writeBatch` (from `writeBatch(db)`) to batch all these writes into a SINGLE network round-trip. This ensures atomic updates and significantly reduces latency.

Let's modify `Trades.tsx` and `ItemDetail.tsx` to use `writeBatch`.
